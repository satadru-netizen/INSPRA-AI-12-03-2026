Discovery
Blueprint agent-ops-blueprint
Transforms discovery call transcripts or project briefs into structured blueprint documents with Figma flowcharts.
Inputs Accepted
Input TypeDescription
Discovery transcriptRaw text from a client discovery call
Step-by-step notesBullet points or structured brief
Client requirements docAny format with project scope
Build Process
  1. Extract discovery data — Pull pain points, goals, target audience, contacts, and tools from the input
  2. Determine applicable phases — Evaluate which of the 8 phases apply (1-6 are core, 7-8 conditional)
  3. Generate blueprint document — Structured markdown with phase breakdown, sub-tasks, dependencies, and timeline
  4. Create Figma flowchart — Visual flow matching the Inspra template style via Figma MCP (Mermaid fallback)
  5. Quality check — Every pain point maps to a goal, every goal maps to a phase, out-of-scope items documented
Output

Markdown blueprint document + Figma flowchart diagram

# [Client Name] — AI [Agent Type] Agent Blueprint Prepared by: Inspra AI Date: [Date] ## Discovery Summary ## Phase 1: Requirements & Knowledge Base ## Phase 2: Data & List Building ## ... (each applicable phase) ## Timeline & Dependencies ## Out of Scope ## Next Steps

"create a blueprint" "scope this project" "here's the discovery call transcript" "new client engagement"

02
Design
Flowchart agent-ops-flowchart
Generates call flow diagrams and SIP networking architecture diagrams for voice agents.
Two Diagram Types
DiagramPurposeWhen Generated
Call FlowchartFull conversation path: greeting → intent → handling → closingAll agents
Networking FlowchartSIP architecture: PSTN → provider → Vapi → agent routingInbound agents only
Build Process
  1. Parse the blueprint or call script — Identify all intents, decision points, and tool interactions
  2. Map conversation paths — Every intent gets a complete path with fallbacks. No dead ends.
  3. Generate in Figma — Color-coded nodes: green (start/end), blue (agent action), yellow (decision), orange (tool call), red (error)
  4. Add networking layer — For inbound: PSTN → SIP trunk → Vapi → agent router with failover paths
Design Rules

Every path has an exit. Max 2 retries on misunderstood intent before human transfer. Post-call automations shown. Decision diamonds labeled with actual conditions. Tool/function call points marked. Failover paths shown as dashed lines.

Trigger Phrases
"create a call flow" "SIP diagram" "map the conversation" "networking flowchart"
03
Build
Prototype agent-ops-prototype
Builds production-ready voice agent prototypes with Vapi or Livekit, following Inspra AI's mandatory system prompt framework.
Build Sequence
  1. Architecture decision — Single agent or squad? Vapi or Livekit? Inbound, outbound, or both?
  2. Model selection — LLM (GPT-4o / Claude Sonnet), voice (matched to brand tone), transcriber (Deepgram Nova-2 default)
  3. System prompt development — Built from the mandatory framework in references/system-prompt-framework.md
  4. Configuration — Uses create-assistant skill for full Vapi setup
  5. Phone number setup — Provision test number via create-phone-number
  6. Demo setup — Test number + web link + instructions for client
System Prompt Framework (Mandatory)

Every Inspra AI agent uses the same core framework. These defaults are non-negotiable:

[Identity] — Name, role, personality (customize) [Context] — Company, scope of calls (customize) [Conversation Principles] — DEFAULT: emotion first, answer first, never interrupt, keep it short, one question at a time [Voice Format Rules] — DEFAULT: dates, times, phone, currency spoken naturally [Conversation Flow] — Step-by-step with decision branches (customize) [Discovery] — DEFAULT: understand before offering [Objection Handling] — DEFAULT: read behind the words, 2 attempts max [Call Ending] — DEFAULT: close with intention, always log outcome [Tools Available] — Never announce tool names (customize) [Hard Compliance] — DEFAULT: no pressure, no fake urgency, recording disclosure, abuse handling, off-topic limits [Boundaries] — DEFAULT + customize per project

Tools agent-ops-tools
Creates Vapi tool definitions, function call schemas, and API integrations that the voice agent calls during conversations.
Common Tool Patterns
ToolTypeIntegrates With
Calendar BookingFunction / API RequestGoogle Calendar, Outlook, Cal.com, GHL
CRM Lookup / UpdateAPI RequestGHL, HubSpot, Salesforce, JobAdder
Transfer CallTransfer Call ToolSIP, phone number, another agent
End CallEnd Call ToolPost-call webhook
Knowledge Base QueryFunctionVector DB, FAQ API
SMS / EmailAPI RequestTwilio, GHL, SendGrid
Custom APIAPI RequestProject-specific endpoints
Design Rules
  1. Name tools clearly — check_available_slots not api_call_1. The LLM decides when to call based on the name.
  2. Write descriptions for the LLM — "Use this tool when the caller wants to book and you have their name and preferred time."
  3. Minimize required params — Only what's truly needed. Fewer params = faster tool calls.
  4. Handle failures — Every tool needs a fallback message: "I'm having trouble checking that right now..."
  5. Test independently — Verify each tool with sample data before connecting to the agent.
Trigger Phrases
"add calendar booking" "create function calls" "set up CRM integration" "add a transfer tool"

Minutes of Meeting agent-ops-mom
Transforms meeting transcripts or raw notes into structured, actionable MoM documents with the Inspra AI format.
Output Format
# Inspra AI x [Client] — [Meeting Type] Date: [Date] | Duration: ~[X] min | Attendees: [Names] ## Executive Summary 3-5 sentences — purpose and outcomes ## Key Decisions Only firm decisions, not discussions ## Action Items | # | Action | Owner | Deadline | Each specific and assignable ## Open Questions Raised but not resolved ## Follow-up Next meeting, agenda, prep required
Quality Rules
  1. Action items are specific — "Tarique to share Drive link by Friday" not "Discuss integration"
  2. Never invent information — If ambiguous, put it in Open Questions
  3. Executive Summary is tight — A busy stakeholder gets the full picture in 30 seconds
  4. Decisions vs discussions — Only confirmed agreements go in Key Decisions
Trigger Phrases
"summarize this meeting" "create MoM" "meeting notes" "what were the action items?"

Build Process
  1. Research first — Always reads n8n docs via MCP before building. Never assumes node configs.
  2. Map the flow — Trigger → nodes → output. One workflow per concern.
  3. Build incrementally — Start with trigger + first node, test, add nodes one at a time.
  4. Error handling — Error branches for every external API call.

09
Post-Live
Prompt Analysis agent-ops-prompt-analysis
Compares system prompts against real call transcripts to find gaps, hallucination risks, tone issues, and missing edge cases.
5-Part Analysis Framework
  1. Transcript Gap Analysis — What the prompt says vs what actually happens. Produces a gap matrix showing handled/mishandled/missing scenarios.
  2. Hallucination Audit — Finds moments where the agent stated information not grounded in the prompt, a tool response, or the caller's words.
  3. Tone Consistency Check — Is tone consistent across caller types? Does it adapt appropriately? Any breaks?
  4. Rule Adherence Check — For each rule in the prompt, verify compliance across all transcripts.
  5. Structural Analysis — Clarity, conflicts, length (<800 words), priority of rules, specificity.
Output

Gap matrix, hallucination flags with transcript references, tone issues with excerpts, rule violation table, structural issues, and a fully revised system prompt with changes highlighted and a change log citing specific transcript evidence.

Revision Rules

Every change must cite a specific transcript. No theoretical improvements. Remove instructions that add no value. Keep under 800 words. Test the revised prompt against the same transcripts that revealed the issues.

Trigger Phrases
"review this prompt" "the agent keeps doing X wrong" "improve the system prompt" "why is the agent saying X?"
